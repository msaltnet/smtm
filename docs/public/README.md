# docs/public — 공유용 프로젝트 문서 작성 가이드

이 폴더는 **다른 프로젝트·팀과 공유하기 위한 요약 문서**를 모아 둔 공간입니다. 내부용 상세 문서(`docs/` 하위의 `architecture/`, `requirements/`, `user-guide/`, `release-notes/` 등)와는 역할이 분리되어 있습니다.

이 가이드는 공유용 문서 6종의 **목적, 대상 독자, 작성 요령**을 정리한 것으로, **다른 프로젝트에서도 동일한 포맷으로 문서를 작성할 수 있도록** 만든 공용 표준입니다.

---

## 1. 이 폴더의 목적

- **프로젝트 간 공유용 요약본 보관**: 외부 팀·프로젝트가 "이 프로젝트가 무엇인지, 어떻게 쓰는지, 어떻게 돌아가는지"를 빠르게 파악할 수 있도록 돕는다.
- **단일 파일 원칙**: 각 주제는 하나의 파일로 정리한다. 상세가 필요하면 내부 문서를 링크로 연결한다.
- **내부 상세 문서와 분리**: 상세 문서는 내부 레퍼런스로 그대로 유지하고, 이곳의 문서는 "공유하기 적당한 수준"으로 요약·정제한다.

---

## 2. 문서 구성 (총 6종)

| 파일 | 역할 | 대상 독자 | 권장 분량 |
|---|---|---|---|
| `overview.md` | 프로젝트가 무엇인지 / 왜 만들었는지 / 핵심 가치·기능·기술 스택 | 모든 독자 (경영/기획/개발/운영) | 100~200줄 |
| `user-guide.md` | 실제로 어떻게 쓰는지 — 주요 시나리오와 흐름 | 실사용자, 신규 온보딩 | 200~400줄 |
| `faq.md` | 자주 묻는 질문과 답변 | 실사용자, 빌더, 운영자 | 200~400줄 |
| `requirements.md` | 제품이 제공해야 하는 기능 스펙 (영역별) | 기획, 개발, QA | 300~500줄 |
| `architecture.md` | 시스템 구조·데이터·런타임·인증 개요 | 개발, SRE, 연동 파트너 | 300~600줄 |
| `release-notes.md` | 버전별 주요 변경 + 향후 로드맵 | 모든 독자 | 200~400줄 |

> 권장 분량은 가이드라인입니다. 짧은 것이 긴 것보다 낫다는 원칙을 지키되, 억지로 분량을 맞추지 마세요.

## 3. 공통 작성 원칙

### 3.1 톤과 스타일
- **평서형·간결한 문장**: "~입니다", "~합니다" 일관성. 한 문장 한 개념.
- **전문용어는 한 번 풀어쓴다**: 약어·프로젝트 고유 용어는 최초 등장 시 괄호 설명.
- **과장·마케팅 표현 배제**: "혁신적", "업계 최고" 같은 수식어 지양. 기능과 사실 중심으로.
- **코드·경로·식별자는 백틱**: `app_config`, `/api/v1/sessions`, `published`.

### 3.2 구조
- **문서 맨 위에 1~3문장 요약**: "이 문서는 …을 다룹니다."
- **목차(ToC) 선택 — 분량이 300줄 넘으면 권장**.
- **H2 섹션 제목은 번호 사용** (예: `## 1. 개요`): 외부에서 참조할 때 안정적.
- **표 적극 활용**: 기능 목록, 상태 전이, 스펙 비교는 표가 글보다 명확.

### 3.3 내부 문서 링크
- **상세가 필요한 지점에 링크**를 남긴다: `[상세 API 스펙](../architecture/01-api-design.md)`.
- 다만 독자가 링크를 따라가지 않아도 개요는 이해할 수 있어야 한다 (공유용 문서의 자족성).

### 3.4 업데이트 정책
- **큰 변화(기능 추가/제거, 아키텍처 변경)만 반영**: 작은 문구 수정은 내부 문서에서 흡수.
- **문서 맨 위에 "최종 갱신일" 또는 기준 버전 명시**.
- `release-notes.md`는 릴리스마다 갱신, 나머지는 필요 시 갱신.

### 3.5 하지 말 것
- 내부 파일 경로, 개인 식별자, 내부 URL 포함 (민감정보 마스킹 원칙).
- 미확정·실험 중 기능을 확정된 기능처럼 기술.
- 내부 상세 문서를 그대로 복사 붙여넣기 — 요약·재구성이 이 폴더의 존재 이유입니다.

## 4. 문서별 작성 요령

### 4.1 overview.md
**목적**: 프로젝트를 처음 보는 사람이 5분 안에 "무엇을, 왜, 어떻게"를 파악하게 한다.

**필수 섹션**
1. **한 줄 요약 / 슬로건** — 프로젝트 정체성
2. **만든 이유 / 해결하는 문제** — 3~5문장
3. **핵심 가치 / 차별점** — 3~5개 불릿
4. **주요 기능** — 도메인 영역별로 그룹핑 (e.g. 탐색·실행·빌드·관리)
5. **기술 스택** — 간단한 표
6. **다음으로 볼 문서** — user-guide / architecture 링크

**피해야 할 함정**: 기능 나열이 너무 상세해지면 user-guide와 중복됨. overview는 "있다"만 보이고, "어떻게 쓰는지"는 user-guide에 넘김.

---

### 4.2 user-guide.md
**목적**: 실사용자가 주요 시나리오를 직접 따라 해볼 수 있는 수준의 안내.

**필수 섹션**
1. **시작하기** — 로그인/접속/권한 필요 조건
2. **주요 사용 흐름** — 역할별(일반 사용자/제작자/관리자)로 각각 번호 리스트 5~10단계
3. **기능별 상세** — 홈, 탐색, 실행, 제작, 관리 등 각 기능 설명
4. **활용 예시 / 시나리오 3~5개** — 구체적 상황 기반

**피해야 할 함정**: UI 캡처 의존. 캡처가 없어도 문장만으로 흐름이 이해돼야 함 (캡처는 시간이 지나면 쉽게 낡음).

---

### 4.3 faq.md
**목적**: 사용 중 자주 막히는 지점을 Q&A로 해소.

**필수 섹션**
1. **일반 FAQ** — 서비스 전반
2. **역할별 FAQ** — 사용자 / 빌더 / 운영자
3. **기능별 FAQ** — 핵심 기능별 (인증, 데이터, 통합 등)
4. **문제 해결 체크리스트** — "X가 안 될 때" 진단 순서

**작성 요령**
- Q는 **실제 사용자가 쓸 법한 구어체**.
- A는 **짧고 직접적으로** — 정책/제약은 근거 숫자까지. (예: "최대 10개", "50,000자")
- **코드 동작 기준**으로 작성 — 가정이나 "예정" 기능은 별도 섹션으로 분리.
- 비슷한 질문은 묶되, 각각 답하기 — 사용자 검색 키워드가 다름.

---

### 4.4 requirements.md
**목적**: 제품이 제공해야 할 기능의 **WHAT** 명세. 읽는 사람이 "이걸 구현해야 하는구나"를 파악.

**필수 섹션**
1. **범위 / 대상 화면** — 어디까지 다루는지
2. **영역별 요구사항** — 도메인별(예: 스토어/빌더/관리자/공통/RAG)로 분리
3. **각 요구사항 표기**
   - `[MVP]` / `[후속]` 구분
   - 간결한 당위 문장: "X는 Y해야 한다"
4. **구현 시 주의사항 / 비기능 요구** — 보안, 성능, 일관성
5. **점검 체크리스트** — QA가 그대로 따라갈 수 있는 수준

**피해야 할 함정**
- 구현 방법(HOW) 기술 — 그건 architecture의 영역.
- 모호한 표현: "빠르게", "사용하기 편하게" — 측정 가능한 기준으로.

---

### 4.5 architecture.md
**목적**: 시스템이 어떻게 구성·동작하는지를 연동 파트너와 신규 엔지니어가 파악.

**필수 섹션**
1. **System Overview** — 컴포넌트 다이어그램 (Mermaid 권장)
2. **기술 스택** — 레이어별 표
3. **Data Model** — 핵심 엔티티 ER 요약
4. **주요 플로우** — 인증, 주요 기능 실행 흐름 (시퀀스 다이어그램)
5. **확장 포인트 / 통합 지점** — API, 외부 시스템 연동, Webhook 등
6. **운영/배포 개요** — 컨테이너, 프록시, 저장소 구성 요약

**작성 요령**
- **Mermaid 다이어그램 활용**: graph, sequenceDiagram, erDiagram.
- 각 섹션은 개요 수준 — 상세는 내부 문서로 링크.
- 컴포넌트 박스 안에 **역할 한 줄**을 넣으면 가독성↑.

**피해야 할 함정**: 구현 세부(함수명, 파일 경로)까지 내려가면 금방 낡음. **책임·계약·흐름** 중심으로.

---

### 4.6 release-notes.md
**목적**: 버전별 변경 사항과 향후 계획 공유.

**필수 섹션**
1. **최신 릴리스 요약 (Headline)** — 이번 릴리스의 3~5개 키포인트
2. **버전별 변경** — 최신 → 과거 역순
   - 각 버전: 날짜 + 테마 + 주요 변경 + 버그 수정
3. **로드맵 / Backlog** — 다음 릴리스 계획과 장기 방향
4. **Breaking Change / 마이그레이션 안내** (있을 때만)

**작성 요령**
- **변경 유형 분류**: 신규 기능 / 개선 / 버그 수정 / 인프라·운영.
- Ref 번호나 이슈 링크가 있으면 괄호로 표기.
- Breaking change는 **눈에 띄게** 표시.

**피해야 할 함정**: 커밋 메시지를 그대로 나열. 독자 관점에서 "무엇이 달라졌는가"로 재구성해야 함.

## 5. 다른 프로젝트에 적용할 때 체크리스트

이 가이드를 다른 프로젝트로 옮길 때:

- [ ] `docs/public/` 폴더 생성 후 이 `README.md`를 먼저 배치
- [ ] 6개 문서를 템플릿 구조로 먼저 만들고, 빈 섹션이라도 제목만 세팅
- [ ] `overview.md`부터 작성 — 나머지 문서의 톤·스타일 기준이 됨
- [ ] 내부 상세 문서가 이미 있다면, **요약·재구성**해서 공유용 버전 생성 (복붙 금지)
- [ ] 민감정보(내부 URL, 자격 증명, 개인식별) 포함 여부 최종 검토
- [ ] 첫 릴리스 이후, 릴리스마다 `release-notes.md` 갱신만이라도 유지

## 6. 디렉터리 구조 예시

```
docs/
├── public/                 # 공유용 요약 (이 폴더)
│   ├── README.md           # 이 가이드
│   ├── overview.md
│   ├── user-guide.md
│   ├── faq.md
│   ├── requirements.md
│   ├── architecture.md
│   └── release-notes.md
├── architecture/           # 내부 상세 (기존 유지)
├── requirements/
├── user-guide/
├── release-notes/
└── ...
```

공유용 문서는 **자족적**이어야 합니다 — 독자가 내부 문서에 접근하지 못하더라도 이 폴더만으로 프로젝트 전반을 이해할 수 있도록 작성하세요.
